# 캘린더 지식 관리 앱 설계안

## 목표

이 앱은 캘린더, 일정 관리, 문서 관리, 통계를 하나로 묶은 로컬 우선 앱이다. 첫 버전은 React + TypeScript 웹 애플리케이션으로 만들고, 처음부터 Tauri 2 기반 데스크탑 및 모바일 앱 패키징을 고려해 설계한다.

MVP는 계정, 서버, 동기화 없이도 충분히 쓸모 있어야 한다. 앱을 열면 오늘 하루를 빠르게 파악할 수 있고, 일정과 관련 문서를 관리하며, 로컬 데이터에서 간단한 인사이트를 확인할 수 있어야 한다.

## 확정 사항

- 기본 시작 화면은 A안인 `오늘 요약`이다.
- 사용자는 설정에서 시작 화면을 변경할 수 있다.
- 설정 기능은 확장 설정 범위로 만든다.
- UI 스타일은 iOS와 비슷한 밝고 정돈된 느낌을 따른다.
- 기술 스택은 React + TypeScript + Tauri 2를 기준으로 한다.
- Tauri 2 패키징 대상은 Windows, macOS, Android, iOS를 염두에 둔다.
- MVP 저장소는 Local Storage를 사용하되, 저장소 추상화 계층을 둔다.
- MVP 범위에서 계정, 서버 백엔드, 클라우드 동기화, 다중 기기 충돌 해결은 제외한다.

## 제품 범위

### 오늘 요약

`오늘 요약`은 앱의 기본 첫 화면이다. 사용자가 하루를 시작하거나 중간에 앱을 열었을 때 바로 필요한 정보를 볼 수 있는 업무판 역할을 한다.

포함 기능:

- 오늘 일정 개수와 다음 일정 표시.
- 오늘 일정의 시간순 목록.
- 빠른 일정 등록.
- 오늘 일정과 연결된 문서 및 메모 표시.
- 자주 만나는 사람, 자주 가는 장소, 오늘의 문서 키워드 같은 작은 인사이트 위젯.

사용자는 설정에서 시작 화면을 바꿀 수 있다. MVP에서 선택 가능한 시작 화면은 `오늘 요약`, `캘린더`, `문서`, `통계`, `설정`이다.

### 캘린더 및 일정 관리

일정 관리 모듈은 구조화된 일정을 관리한다.

일정의 기본 필드:

- 제목.
- 분류.
- 참여자.
- 시작 시간과 종료 시간.
- 장소.
- 연결 페이지 또는 참고 링크.
- 설명 또는 메모.
- 연결 문서.
- 생성일과 수정일.

캘린더 보기는 MVP에서 월간 보기와 일간 보기부터 시작한다. 주간 보기는 일정 생성과 조회 흐름이 안정된 뒤 확장한다.

### 문서 관리

문서는 일정에 연결될 수 있고, 독립 문서로도 존재할 수 있다. MVP 문서 기능은 일정과 지식 기록을 연결하는 데 초점을 둔다.

포함 기능:

- 일정에 대한 메모 또는 문서 작성.
- 기본 블록 에디터 구조.
- 이미지, 도표, 파일 업로드를 위한 첨부 메타데이터.
- 사용자가 편집할 수 있는 문서 요약 필드.
- 문서 키워드.
- 문서 분류.
- 작성자.
- 생성일과 수정일.

MVP에서는 실제 바이너리 파일 저장을 제한적으로 다룬다. 다만 문서 모델에는 첨부 메타데이터를 포함해, 이후 Tauri 파일 API를 붙이더라도 문서 모델 자체를 크게 바꾸지 않도록 한다.

### 통계 대시보드

통계 모듈은 일정과 문서 데이터를 읽어 로컬에서 인사이트를 계산한다. 통계 데이터는 사용자가 직접 편집하는 데이터가 아니라, 일정과 문서에서 파생되는 읽기 모델로 취급한다.

포함 기능:

- 분류별 일정 시간 통계.
- 주된 만남 또는 자주 만나는 참여자.
- 자주 가는 장소.
- 문서 키워드 빈도.
- 분류별 주요 문서 주제.
- 문서 키워드 연결 그래프를 만들기 위한 데이터.

MVP의 통계는 클라이언트에서 계산한다. 문서 머신 러닝 분류는 향후 기능으로 남기며, 첫 출시를 막는 필수 기능으로 두지 않는다.

### 확장 설정

설정 화면은 iOS의 그룹형 설정 화면과 비슷한 형태로 만든다. 각 설정 항목은 현재 값을 오른쪽에 보여주고, 세부 화면 또는 선택 모달로 들어갈 수 있게 한다.

MVP 설정 항목:

- 시작 화면 선택.
- 캘린더 기본 보기 선택.
- 문서 요약 옵션.
- 기본 문서 분류 방식.
- 통계 표시 옵션.
- 테마: 시스템, 라이트, 다크.
- 로컬 백업 및 가져오기 진입점.

모든 설정은 로컬에 저장된다. 앱 시작 시 설정을 먼저 읽고, `시작 화면` 값에 따라 초기 라우트를 결정한다.

## 아키텍처

앱은 기능 모듈과 공통 플랫폼 서비스를 분리하는 구조로 만든다. 핵심 목표는 기능이 늘어나도 특정 화면이나 저장 방식에 코드가 묶이지 않게 하는 것이다.

```text
src/
  app/
    App.tsx
    routes.tsx
    providers/
  features/
    today/
    calendar/
    schedules/
    documents/
    statistics/
    settings/
  shared/
    components/
    design-system/
    storage/
    dates/
    types/
    utils/
  platform/
    tauri/
    web/
```

### 계층 구조

React 화면 컴포넌트는 Local Storage를 직접 읽거나 쓰지 않는다. 화면은 기능별 훅이나 서비스를 호출하고, 기능 서비스는 공통 저장소 어댑터를 호출한다. 이렇게 하면 나중에 IndexedDB, Tauri 기반 SQLite, 암호화된 로컬 파일 저장소로 옮기더라도 화면 코드를 크게 바꾸지 않을 수 있다.

권장 의존 방향:

```text
UI 화면 -> 기능 훅/서비스 -> 저장소 인터페이스 -> 저장소 구현체
```

Tauri 전용 API는 `src/platform/tauri` 아래에 둔다. React 기능 모듈은 일반 브라우저 개발 환경에서도 동작해야 한다.

### 저장소 인터페이스

저장소 계층은 타입이 있는 저장소 객체를 제공한다.

- `settingsRepository`
- `scheduleRepository`
- `documentRepository`
- `attachmentRepository`

각 저장소는 기본적인 생성, 조회, 수정, 삭제, 목록 조회를 제공한다. 저장 키에는 버전을 포함해 향후 마이그레이션을 추가할 수 있게 한다.

저장 키 예시:

```text
calendarApp:v1:settings
calendarApp:v1:schedules
calendarApp:v1:documents
calendarApp:v1:attachments
```

### 데이터 모델

핵심 엔티티:

- `AppSettings`
- `Schedule`
- `Document`
- `Attachment`
- `Category`
- `Participant`
- `StatisticSnapshot`

관계:

- 하나의 일정은 여러 문서와 연결될 수 있다.
- 하나의 문서는 MVP에서 0개 또는 1개의 일정과 연결될 수 있다.
- 첨부 항목은 문서에 속한다.
- 통계는 일정과 문서에서 계산되는 파생 데이터이며 직접 편집하지 않는다.

### 확장성 원칙

아키텍처를 계속 확장할 수 있게 다음 원칙을 따른다.

- 새로운 제품 영역은 `src/features` 아래 기능 모듈로 추가한다.
- 여러 기능에서 쓰는 UI는 `src/shared/components`에 둔다.
- 도메인에 묶이지 않는 유틸리티는 `src/shared`에 둔다.
- OS 또는 앱 패키징 관련 코드는 `src/platform` 뒤에 숨긴다.
- 새로운 저장 방식을 추가하기 전에 저장소 인터페이스를 먼저 정의한다.
- 저장 데이터에는 버전을 두고, 스키마가 바뀌면 마이그레이션을 작성한다.
- 통계는 파생 읽기 모델로 다뤄 새로운 인사이트를 추가해도 일정과 문서 편집 흐름이 흔들리지 않게 한다.

## UI 방향

UI는 iOS 디자인 패턴에서 영감을 받되, Apple의 고유 자산을 복제하지 않는다. 목표는 밝고 조용하며 반복 사용에 편한 생산성 앱이다.

스타일 원칙:

- 시스템 그룹 배경과 비슷한 밝은 중립 배경.
- 흰색 카드와 그룹형 설정 섹션.
- 적당한 둥근 모서리와 절제된 그림자.
- 주요 액션과 선택 상태에는 파란색 포인트 컬러 사용.
- 상태나 인사이트에는 초록색, 주황색 등 보조 색을 제한적으로 사용.
- 모바일에서는 하단 내비게이션 사용.
- 데스크탑에서는 사이드바 또는 상단 내비게이션 사용.
- 설정 화면은 오른쪽에 현재 값을 보여주는 그룹형 리스트를 사용.

첫 화면은 매일 반복해서 쓰는 업무 화면이므로 너무 홍보 페이지처럼 보이면 안 된다. 큰 히어로 영역, 과한 장식 카드, 불필요한 설명 문구는 피한다.

## 라우팅

주요 라우트:

- `/today`
- `/calendar`
- `/schedules/:id`
- `/documents`
- `/documents/:id`
- `/statistics`
- `/settings`

앱 시작 시 `AppSettings.startScreen` 값을 읽고 해당 화면으로 이동한다. 설정값이 없거나 잘못된 값이면 `/today`로 이동한다.

## 오류 처리

MVP 오류는 로컬 앱의 사용성을 해치지 않는 방식으로 처리한다.

- 저장된 데이터 형식이 잘못되면 안전한 기본값으로 복구하고 복구 가능한 오류로 기록한다.
- 저장 실패는 해당 액션 근처에 인라인 오류로 보여준다.
- 가져오기 실패는 파일을 읽거나 해석할 수 없었다고 명확히 안내한다.
- 연결된 일정이나 문서가 삭제된 경우 화면이 깨지지 않도록 `삭제됨` 또는 `사용 불가` 상태를 보여준다.

Local Storage에서 읽은 값은 사용하기 전에 구조를 검증한다.

## 테스트 전략

초기 테스트는 로컬 데이터와 라우팅을 보호하는 동작에 집중한다.

- 설정 기본값과 시작 화면 라우팅.
- 일정 생성, 수정, 삭제, 목록 조회.
- 문서 생성과 일정 연결.
- 샘플 데이터 기반 통계 계산.
- 저장소 마이그레이션과 잘못된 데이터 복구.

UI 구현 후에는 데스크탑과 모바일 화면 폭에서 시각 검증을 수행한다.

## MVP 구현 순서

1. React + TypeScript 앱을 생성하고 라우팅과 iOS 스타일 기본 CSS를 만든다.
2. 타입 모델과 Local Storage 저장소를 만든다.
3. 시작 화면과 테마를 저장할 수 있는 설정 기능을 먼저 만든다.
4. 샘플 데이터 기반 `오늘 요약` 화면을 만들고, 이후 실제 로컬 데이터에 연결한다.
5. 일정 CRUD와 캘린더 보기를 만든다.
6. 문서 관리와 일정-문서 연결을 만든다.
7. 로컬 데이터에서 계산하는 통계 대시보드를 만든다.
8. Tauri 설정을 추가하고 패키징 전제를 검증한다.

## 향후 확장

이 설계는 다음 확장을 염두에 둔다.

- Tauri 기반 파일 저장과 첨부 파일 관리.
- SQLite 또는 IndexedDB 저장소.
- 데스크탑 파일 드래그 앤 드롭.
- 모바일 카메라 또는 파일 선택기 연동.
- 일정과 문서 전체 검색.
- 오프라인 내보내기와 가져오기.
- MVP 이후 선택적 동기화 서비스.
- 문서 분류와 키워드 인사이트를 위한 머신 러닝 기능.
- 문서 키워드 관계를 보여주는 고급 그래프 시각화.

## 구현 참고

MVP는 고급 AI 기능이나 동기화보다, 제대로 동작하는 로컬 우선 앱을 우선한다. 문서 요약과 키워드 추출은 처음에는 사용자가 편집 가능한 필드나 간단한 로컬 규칙으로 시작하고, 문서 작성 흐름이 안정된 뒤 AI 보조 기능으로 확장한다.
